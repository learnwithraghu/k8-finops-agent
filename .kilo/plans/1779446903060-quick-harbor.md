# K8s FinOps Agent - Implementation Plan

## Overview
Build an AI-powered K8s FinOps agent that scans Kubernetes resources for missing tags and unused resources, then uses AWS Bedrock as the decision layer to classify issues, recommend ownership, and generate actionable records. For the prototype, the agent will send decisions to a simple Flask-based issue tracker UI instead of relying only on GitHub Issues.

## Architecture

### Components
1. **K8s Scanner** - Python script using kubernetes-client to query cluster resources
2. **Bedrock Decision Layer** - AWS Bedrock prompts for classification, ownership, and issue drafting
3. **Flask Issue Tracker** - Simple internal UI/API to review and manage findings
4. **Optional GitHub Sync** - Export selected issues to a tech-debt repository
5. **Demo Resources** - Sample airline booking microservices with intentional violations

### Resource Scope
- Deployments, Services, PVCs, ConfigMaps
- Pods/Ingress can be added later if needed
- Checks for both AWS-style tags and Kubernetes standard labels

### Tagging Convention (Airline Booking Context)
Based on an airline booking system organizational structure:
- `owner` / `app.kubernetes.io/owner` - Team/individual responsible
- `environment` / `app.kubernetes.io/env` - dev/staging/prod
- `cost-center` / `app.kubernetes.io/cost-center` - Budget allocation (e.g., "booking-engine", "payment", "inventory")
- `application` / `app.kubernetes.io/name` - Service name (e.g., "flight-search", "booking-api")
- `tier` / `app.kubernetes.io/tier` - frontend/backend/database/cache
- `compliance` / `compliance-level` - PCI, SOX, etc.

## Infrastructure Context

### EC2 Instance Specifications
- **Instance Type**: t2.medium (2 vCPU, 4 GB RAM)
- **OS**: Amazon Linux 2
- **K8s Setup**: Docker + Helm + Kubernetes (single-node or minimal cluster)
- **Resource Constraints**: All K8s resources must be sized appropriately for t2.medium limits

### Resource Sizing Guidelines
- Pod memory limits: 256Mi - 512Mi per pod
- CPU limits: 250m - 500m per pod
- Replicas: 1-2 max per deployment (to fit within 4GB)
- Avoid resource-heavy databases; use lightweight alternatives

2. **Cost Calculation Engine** (`cost_calculator.py`):
   - Base calculation: t2.medium hourly rate (~$0.0464/hour)
   - Resource allocation formula:
     ```
     pod_cost_per_hour = (instance_hourly_rate / total_cluster_cpu) * pod_cpu_request
     monthly_cost = pod_cost_per_hour * hours_in_month * replicas
     ```
   - Factors in:
     - CPU requests/limits
     - Memory requests/limits
     - Storage (PVC) costs
     - Network egress (estimated)

3. **Untracked Money Detection** (`untracked_money.py`):
   
   **Logic for "Untracked Money"**:
   ```python
   def calculate_untracked_money(resource):
       # Scenario 1: Missing cost-center tag
       if not has_cost_center_tag(resource):
           return calculate_monthly_cost(resource)  # Full cost is untracked
       
       # Scenario 2: Cost-center exists but no owner
       if has_cost_center_tag(resource) and not has_owner_tag(resource):
           return calculate_monthly_cost(resource) * 0.5  # Partially tracked
       
       # Scenario 3: Orphaned resources (PVCs not mounted)
       if is_orphaned_pvc(resource):
           return calculate_storage_cost(resource)  # 100% waste
       
       # Scenario 4: Resources in "default" namespace with no labels
       if resource.namespace == "default" and not resource.labels:
           return calculate_monthly_cost(resource)  # Likely untracked
       
       return 0  # Properly tracked
   ```
   
   **Untracked Categories**:
   - **Unallocated**: Missing cost-center (cannot bill to any team)
   - **Orphaned**: Resources not actively used (PVCs, old ConfigMaps)
   - **Unowned**: Has cost-center but no owner (no accountability)
   - **Unknown**: In default namespace with minimal labels

## Cost Calculation Logic (Dummy Logic for Demo)

```python
# config/pricing.yaml
pricing:
  ec2:
    t2.medium:
      hourly: 0.0464
      vcpu: 2
      memory_gb: 4
  storage:
    gp3:
      gb_per_month: 0.08
  
# Calculation for t2.medium cluster
def calculate_resource_cost(resource):
    instance_hourly = 0.0464
    hours_per_month = 730
    
    # Pod cost based on CPU request
    cpu_fraction = resource.cpu_request / 2000  # 2 vCPUs
    pod_hourly = instance_hourly * cpu_fraction
    pod_monthly = pod_hourly * hours_per_month * resource.replicas
    
    # Storage cost
    if resource.pvc:
        storage_monthly = resource.pvc.size_gb * 0.08
        pod_monthly += storage_monthly
    
    return pod_monthly
```
