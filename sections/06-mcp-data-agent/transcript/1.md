### Opening (~20 sec)

Welcome to Section 06. In Section 05 you proved the read path with curl — health check, handshake, then a real cluster read through MCP.

That worked. But the setup had two moving parts: the Kubernetes MCP server inside Docker, and Supergateway translating its stdio protocol into HTTP so curl could reach it.

Today we simplify the stack for Python agents.

> **Visual:** Side-by-side diagram — Section 05: MCP container → Supergateway → client. Section 06: MCP container → client.

### Two ways to get HTTP MCP (~35 sec)

Both paths expose the same thing: kubectl as MCP tools over HTTP. Same protocol. Same tool names — `kubectl_get`, `kubectl_describe`, and the rest. Different packaging.

**The Section 05 path** runs the MCP server over stdio and wraps it with Supergateway. That bridge is perfect when you need HTTP quickly and the server has no native HTTP mode — curl validation, smoke tests, teaching the protocol.

**The Section 06 path** runs the MCP server with native Streamable HTTP built in. One Docker container. One port. No gateway process in the middle.

> **Visual:** Highlight "extra process" on the Supergateway side; fade it out on the standalone side.

> **Say:** "Same tools on the cluster side. We just drop the translator when the server already speaks HTTP."

---

### Why switch for agents (~25 sec)

Curl needed a stable HTTP endpoint. LangChain agents need the same — a connection they can call repeatedly while the ReAct loop runs.

A persistent container with native HTTP is simpler to start, simpler to explain, and one fewer thing that can fail mid-demo.

You still mount kubeconfig read-only. You still restrict demos to read-only tools. The security model does not change — only the wiring.
