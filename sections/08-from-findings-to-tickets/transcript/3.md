# Video 6: Architecture Update — The Closed-Loop FinOps Pipeline

**Duration:** 3–4 minutes  
**Slides:** `slides/slide5_closed_loop_pipeline.svg`

---

Let's update the architecture diagram with everything we have built so far.

**Section 05 — MCP Setup.** Expose the Kubernetes MCP server. Read-only cluster access validated with curl or Python.

**Section 06 — Collect.** LangChain agents call MCP tools and produce unstructured cluster reports. Output goes to stdout — findings exist but are not actionable yet.

**Section 07 — Analyze with policy.** Tagging rules from YAML constrain what the LLM flags. Same agent pattern, but grounded in FinOps policy rather than ad-hoc judgment.

**Section 08 — Act (this section).** Two halves in one lesson:
- **Tracker service** — Kanban board, REST API, MCP server
- **Agent integration** — unified LangChain agent with K8s + tracker MCP tools

The loop is now closed: cluster → findings → tickets → human triage.

**Section 09 — Refactor.** The integration agent is intentionally small (~4 files), but as we add retries, deduplication, and scheduling, we split into collector / analyzer / tracker modules. Separation of concerns for production code.

**Section 10 — Deploy.** Package the agent as a Helm chart. Run periodic scans inside Kubernetes as a CronJob.

The key insight: each section adds one capability without rewriting the previous ones. Section 06's `mcp_client.py` survives unchanged into Section 08. The tracker service from Demo 1 is the same destination the agent writes to in Demo 3.

FinOps AI is not one monolithic agent — it is a pipeline of composable pieces. Collect, analyze, act, refactor, deploy. That is the architecture we will carry into production.
