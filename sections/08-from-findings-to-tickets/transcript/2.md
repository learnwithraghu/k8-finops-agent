# Video 4: Whiteboard — Wiring the MCP Agent to the Tracker

**Duration:** 3–4 minutes  
**Slides:** `slides/slide4_integration_whiteboard.svg`

---

We have two separate systems: the K8s MCP agent from Section 06 and the issue tracker we just built. The whiteboard question is — how do we connect them with minimal change?

**Keep collection unchanged.** The Section 06 agent already knows how to call `kubectl_get` via LangChain and MCP. We do not rewrite collection. We copy `mcp_client.py` as-is.

**Add a structure step.** The agent returns plain English — not JSON tickets. A small `structure.py` module sends the audit text to the LLM with a strict schema: one `TrackerTicket` per finding. Fields match `IssueCreate` on the tracker side: title, namespace, resource_name, priority, reasoning.

**Post via tracker MCP.** `tracker_client.py` connects to port 8086 and calls `create_issue` for each ticket. Same payload the manual curl used in Demo 2 — just automated.

The data flow:

```
K8s MCP (:8000) → run_agent() → audit text
audit text → structure_findings() → TicketBatch
TicketBatch → tracker MCP (:8086) → Kanban board (:8085)
```

**Payload contract:** `TrackerTicket` on the agent side maps 1:1 to `IssueCreate` on the tracker side. Title format like `[FinOps] namespace/resource - CATEGORY`. Priority normalized to critical / high / medium / low. Reasoning field carries the LLM's explanation for human triage.

**What we deliberately skip:** Re-implementing cluster collection in the integration agent. That was the old Section 09 monolith problem — one file doing collect, analyze, and post. We keep concerns separated: Section 06 collects, Section 07 supplies policy context, Section 08 structures and posts.
