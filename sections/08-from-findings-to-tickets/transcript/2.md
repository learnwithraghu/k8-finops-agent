# Video 4: Whiteboard — Wiring the MCP Agent to the Tracker

**Duration:** 3–4 minutes  
**Slides:** `slides/slide4_integration_whiteboard.svg`

---

We have two MCP servers: Kubernetes for cluster access and the issue tracker for ticket creation. The whiteboard question is — how does one agent use both?

**One LangChain agent, two MCP servers.** We extend `MCP_SERVERS` in `mcp_client.py` to register both the K8s MCP (Streamable HTTP on :8000) and the tracker MCP (SSE on :8086). `convert_mcp_to_langchain_tools` discovers tools from both and gives the agent one combined tool list: `kubectl_get`, `create_issue`, `list_issues`, and so on.

**One ReAct loop does everything.** The agent prompt tells the LLM to audit cluster labels and call `create_issue` for each finding. No separate structure step, no imperative posting loop. The LLM decides when to fetch data and when to create tickets — same pattern as Section 06, just with more tools.

The data flow:

```
LangChain agent
  ├── K8s MCP (:8000) → kubectl_get, etc.
  └── Tracker MCP (:8086) → create_issue, list_issues
         └── Kanban board (:8085)
```

**Payload contract:** Each `create_issue` call uses the same fields as the manual curl demo: title, namespace, resource_name, priority, reasoning. Tagging rules from Section 07 arrive via `SystemMessage`.

**Why not a separate posting client?** If the tracker already exposes MCP tools, LangChain should call them the same way it calls K8s tools. A separate `tracker_client.py` bypasses the agent loop — useful for deterministic batch posting, but not needed when the goal is a fully agent-based pipeline.
