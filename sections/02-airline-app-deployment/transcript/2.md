# Video: Manifest bundle walkthrough — one apply, many namespaces

**Time Budget:** 3 min

### Navigation

Open these files in order as you record. You do not need every YAML in the tree — teach the pattern once, then contrast.

```text
sections/02-airline-app-deployment/manifests/airline-k8-deployment/
├── kustomization.yaml                    ← start here
├── flight-search-service/
│   ├── kustomization.yaml                ← the repeating pattern
│   ├── deployment.yaml                   ← workload + labels
│   ├── service.yaml                      ← stable DNS (glance only)
│   ├── configmap.yaml                    ← config slot (glance only)
│   └── pvc.yaml                          ← storage slot (glance only)
├── inventory-service/deployment.yaml       ← contrast: well-tagged
├── analytics-service/deployment.yaml     ← contrast: wrong namespace
└── problem-resources/
    └── orphaned-pvc.yaml                 ← foreshadow Section 03
```

---

### Opening

Before you run `kubectl apply`, spend three minutes here. The manifests folder looks busy — six subfolders, dozens of YAML files. You do not need to read them all. Six blocks explain the whole deployment.

---

### Block 1 — Root Kustomize entry point

**Open:** `manifests/airline-k8-deployment/kustomization.yaml`

Scroll to the `resources:` list. Five service folders plus `problem-resources`. That is the entire airline footprint. Kustomize is built into kubectl — no Helm, no templating engine. One `kubectl apply -k` on this directory creates Deployments, Services, ConfigMaps, and PVCs across multiple namespaces. **Aha:** one file, many namespaces.

---

### Block 2 — The per-service pattern

**Open:** `flight-search-service/kustomization.yaml`

Two things jump out. First, `namespace: flight-search` — Kustomize stamps every resource in this folder into that namespace. Second, four files: `deployment.yaml`, `service.yaml`, `configmap.yaml`, `pvc.yaml`. Every service folder repeats this shape. **Aha:** a microservice in Kubernetes is not just a Deployment — it is a bundle of four primitives.

Glance at `service.yaml`, `configmap.yaml`, and `pvc.yaml` in the same folder. Service gives stable DNS. ConfigMap holds configuration without rebuilding an image. PVC requests persistent storage. Same slots in every service — we will not open all five copies.

---

### Block 3 — The workload itself

**Open:** `flight-search-service/deployment.yaml`

Do not read line by line. Jump to three spots: `image: nginx:alpine`, the `containerPort`, and `resources` requests and limits. The image is a stand-in. The port and resource requests are real — they are what the scheduler uses and what FinOps will eventually care about. **Aha:** we are learning topology and operations, not airline code.

Notice the labels — `environment: prod` is there, but comments call out missing `owner` and `cost-center`. File that away; we will come back to it in Section 03.

---

### Block 4 — Same shape, different maturity

**Open:** `inventory-service/deployment.yaml` — keep `flight-search-service/deployment.yaml` visible for comparison.

Same Deployment structure. But inventory carries `owner`, `cost-center`, and `criticality`. Flight search does not. **Aha:** identical YAML layout, different governance maturity. That gap is realistic — and it is why FinOps exists.

---

### Block 5 — The misplaced workload

**Open:** `analytics-service/deployment.yaml`

Look at `metadata.namespace` — `default`, not a team namespace. Minimal labels. **Aha:** not every workload lands where it should. A cluster can look healthy and still have hygiene problems hiding in plain sight.

---

### Block 6 — Waste baked into a clean deploy

**Open:** `problem-resources/orphaned-pvc.yaml`

Read the comment at the top: a 5Gi PVC in the `airline` namespace, not mounted by any pod. It deploys alongside the services because `problem-resources` is in the root kustomization. **Aha:** a successful `apply` can already contain latent waste. Section 03 will hunt for exactly this.

---

### Mental model

Tie it together: Kustomize root → `kubectl apply -k` → namespace → Deployment creates Pods, Service gives DNS, ConfigMap and PVC sit alongside as separate concerns. Deployment is desired state. Pod is what you inspect on-call. Everything else supports the workload.

---

### Close

That is the whole bundle — root Kustomize, per-service folder pattern, nginx stand-ins, intentional label gaps, and problem resources hiding in the same deploy.
