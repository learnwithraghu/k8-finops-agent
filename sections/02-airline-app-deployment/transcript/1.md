# Video: SkyLine Air services → namespaces

**Time Budget:** 3 min

### Opening

Welcome back. In Section 01 you built a Kind cluster — `finops-cluster` — and carved out team namespaces: `booking-api`, `flight-search`, `inventory`, `payment`, and `airline`. They were empty boxes. Now we fill them.

SkyLine Air runs as a set of microservices. In this section we deploy five of them. Each one maps to a real team and a real slice of the business.

**booking-api** handles reservations — creating and confirming bookings.
**flight-search-service** powers search and availability — what flights can you actually book?
**inventory-service** tracks seats and routes — what capacity is left?
**payment-processor** charges cards when a booking goes through.
**analytics-collector** gathers usage telemetry so the business can see what customers do.

Four of these land in their team namespaces — `booking-api`, `flight-search`, `inventory`, and `payment`. That is deliberate: each team owns its box. The fifth — analytics — sits in the `default` namespace. That is not an accident either. Real clusters have workloads that never got assigned to a proper home. We will see that choice when we open the manifests.

---

### What we are actually deploying

Here is the important twist: there is no airline application source code in this repo. No Go services, no Dockerfiles to build. The "app" is a **manifest bundle** — YAML files that tell Kubernetes what to run.

Inside each Deployment you will find `nginx:alpine` — a tiny, fast-pulling web server standing in for the real microservice. We are not teaching airline business logic. We are teaching **Kubernetes operations**: deploy, inspect, describe, read logs, exec into a container. The cluster shape is realistic; the container image is a safe shortcut.

If you have `architecture.svg` open in this section folder, use it as your map. Each box is a namespace. Inside each box you will eventually see a Deployment, a Service, a ConfigMap, and a PVC — the four primitives that make up almost every production workload.

---

### What comes next

In the live demos you will run three beats. First, one `kubectl apply -k` command deploys the entire footprint across multiple namespaces. Second, you validate that pods are Running, services exist, and PVCs are Bound. Third, you triage a single service — `flight-search-service` — using describe, logs, and exec. That last flow is what on-call engineers do every day.

### Close
