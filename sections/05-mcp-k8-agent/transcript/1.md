# Video: Demo — Start the MCP Gateway (Supergateway Exposes kubectl as HTTP)

**Time Budget:** 3 min

> **Legend:** Lines in blockquotes are presenter cues — actions and timing. Do not read them aloud. Everything else is your script.

---

### Opening (~30 sec)

Welcome to Section 05. In Section 04 we designed the Collect stage — Gather platform state into neutral inventory. Today we build the read path.

We have a prebuilt MCP server that wraps kubectl. Supergateway exposes it as HTTP so we can test with curl — no SDK, no client library, just raw requests. This is the fastest way to prove MCP works before we plug in a Python agent in Section 06.

> **Do:** Open a terminal. Confirm you are in the project root.

---

### Step 1 — Validate local cluster access (~30 sec)

Before we touch MCP, confirm kubectl works. The MCP server uses the same kubeconfig. If kubectl fails, MCP will fail.

> **Do:** Run `kubectl config current-context`, then `kubectl get ns`.

You should see `kind-finops-cluster` and the airline namespaces — `booking-api`, `flight-search`, `inventory`, `payment`. Good. The cluster is reachable and the MCP server will inherit this access.

---

### Step 2 — Confirm the kubeconfig path (~20 sec)

The MCP server runs inside Docker. It needs your kubeconfig mounted as a volume. Let us resolve and validate the path first.

> **Do:** Run the `KUBECONFIG_FILE` export and `test -f` check.

You should see `Using ~/.kube/config` — or whatever path your environment resolves. Kind writes there by default. We mount it read-only into the container so the MCP server can authenticate without copying credentials.

---

### Step 3 — Start the MCP HTTP endpoint (~90 sec)

Now the main event. Open a **dedicated terminal** for this — Supergateway stays running for the whole session.

> **Do:** Run the full `npx supergateway` command. Read it aloud layer by layer as it starts.

Walk through the pieces:

- `docker run mcp/kubernetes:latest` — the prebuilt MCP server, running kubectl as tools inside a container.
- `-v ${KUBECONFIG_FILE}:/kubeconfig:ro` — kubeconfig mounted read-only.
- `-e KUBECONFIG=/kubeconfig` — tells kubectl inside the container where to look.
- `--network host` — so the container can reach the Kind API server on localhost.
- `supergateway --outputTransport sse --port 8000` — converts the server's stdio protocol into HTTP/SSE on port 8000.
- `--healthEndpoint /healthz` — a liveness probe any client can hit.

> **Do:** Wait for the expected output.

You should see **"Listening on port 8000"**. Leave this terminal open. Do not close it — Demo 2 needs it running.

Supergateway is the bridge: stdio for local tools, HTTP for remote clients. Every agent in Sections 06 and 07 will talk to this same endpoint.

---

### Close (~10 sec)

The MCP gateway is live on port 8000. Next we prove the whole chain works with three curl calls — health, handshake, and a real cluster read. Open `2_guide.md`.
