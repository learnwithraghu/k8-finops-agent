# Video: Demo — Validate the MCP-to-Kubernetes Connection with curl

**Time Budget:** 3–4 min

> **Legend:** Lines in blockquotes are presenter cues — actions and timing. Do not read them aloud. Everything else is your script.

---

### Opening (~20 sec)

Supergateway is running in another terminal. Now we prove the chain with raw HTTP — no Python, no SDK, no client library. If curl works, any MCP client works.

> **Do:** Open a **new** terminal. Supergateway stays running in the other one.

---

### Step 1 — Health probe (~20 sec)

> **Do:** Run `curl -s http://localhost:8000/healthz`.

Expected: `ok`. The HTTP layer is up.

---

### Step 2 — MCP initialize (~40 sec)

> **Say:** "Streamable HTTP means request and response travel on one HTTP call — curl in, JSON out."

> **Do:** POST `initialize` to `http://localhost:8000/mcp`.

Expected: a reply in the same curl with server name **"kubernetes"**. No second terminal, no session id.

This is the protocol handshake — not kubectl yet.

---

### Step 3 — List namespaces (~40 sec)

> **Do:** POST `tools/call` for `kubectl_get` / namespaces to the same `/mcp` endpoint.

Expected: airline namespaces — `booking-api`, `flight-search`, `inventory`, `payment`.

We just read the cluster through MCP with curl. That is the Gather seam Section 06 will automate.

---

### Step 4 — Cleanup (~15 sec)

> **Do:** `Ctrl-C` Supergateway when the session is over.

---

### Close (~15 sec)

MCP endpoint validated. Next: Section 06 builds a Python agent against this same endpoint.
